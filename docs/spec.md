# 游戏测试平台 - 规格说明书

## 1. 项目概述

### 项目名称
游戏测试平台 (Game Testing Platform)

### 项目类型
Tauri 桌面应用（开源项目）

### 核心功能
游戏测试Issue管理与工作流程跟踪，支持本地/服务器双模式

### 目标用户
- 游戏测试人员
- 开发人员
- 项目管理人员

## 3. 状态系统

### 3.1 Issue状态

| 状态 | 颜色 | 说明 |
|------|------|------|
| Pending | 灰色 (#9E9E9E) | 待处理 |
| InProgress | 蓝色 (#2196F3) | 处理中 |
| Fixed | 橙色 (#FF9800) | 已修复 |
| Verified | 绿色 (#4CAF50) | 已验证 |

### 3.2 优先级等级

| 优先级 | 颜色 | 图标 | 说明 | 响应时间 |
|--------|------|------|------|----------|
| P1-紧急 | 红色 (#FF0000) | 🔴 | 系统崩溃、数据丢失 | < 1小时 |
| P2-高 | 橙色 (#FF6B00) | 🟠 | 功能无法使用 | < 4小时 |
| P3-中 | 黄色 (#FFD700) | 🟡 | 功能受限但有替代方案 | < 24小时 |
| P4-低 | 绿色 (#00FF00) | 🟢 | 小问题、优化建议 | < 72小时 |

## 5. 人员系统

### 5.1 用户角色

| 角色 | 显示名称 | 权限 | 说明 |
|------|----------|------|------|
| admin | 管理员 | 全部权限 | 系统管理员，可管理成员、标签、配置 |
| developer | 开发 | 创建/修改/修复Issue | 开发人员，处理Issue |
| tester | 测试 | 创建/验证Issue | 测试人员，提交和验证Issue |

### 5.2 管理员机制

```
管理员职责:
- 用户管理: 添加/删除/修改用户角色
- 标签管理: 创建/编辑/删除标签
- Issue管理: 可操作所有Issue（不受所有权限制）
- 系统配置: 服务器连接、同步设置
- 数据导出: 导出全部数据备份

管理员权限:
- 指派任意Issue的负责人
- 转交任意Issue给其他开发人员
- 删除任意Issue
- 强制关闭/重新打开Issue
- 查看所有统计数据
```

### 5.3 请求人/负责人逻辑

```
请求人 (requester): 提交Issue的人，显示格式: 姓名(岗位)
负责人 (assignee): 被指派处理Issue的人，显示格式: 姓名(岗位)

岗位显示对照:
- admin → 管理员
- developer → 开发
- tester → 测试

规则:
- 请求人默认为Issue创建者
- 负责人可由管理员或请求人指派
- 负责人可以转交给其他开发人员（需管理员批准）
- Issue状态变更需要负责人确认
- 管理员可强制变更任意Issue状态
```

### 5.4 人员数据模型

```typescript
interface User {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'developer' | 'tester';
  roleDisplay: string;  // 显示名称: 管理员/开发/测试
  avatarUrl?: string;
  stats: {
    issuesCreated: number;    // 创建的Issue数
    issuesAssigned: number;   // 被指派的Issue数
    issuesCompleted: number;  // 完成的Issue数
    issuesVerified: number;   // 验证的Issue数
  };
  createdAt: string;
}

// 获取显示名称
function getRoleDisplay(role: string): string {
  switch (role) {
    case 'admin': return '管理员';
    case 'developer': return '开发';
    case 'tester': return '测试';
    default: return role;
  }
}

// 格式化用户显示
function formatUserDisplay(user: User): string {
  return `${user.name}(${user.roleDisplay})`;
}
```

## 6. Issue状态流转

### 6.1 流程图

```
测试人员测试 → 提交Issue → 人员工单回复 → 开始修复 → 修复完成 → 测试人员确认修复完成
     ↓              ↓              ↓             ↓            ↓              ↓
   无Issue        Pending → InProgress → Fixed → Verified
```

### 6.2 状态变更规则

| 当前状态 | 可变更状态 | 操作角色 | 说明 |
|----------|-----------|----------|------|
| Pending | InProgress | developer/admin | 开发人员点击"开始修复" |
| InProgress | Fixed | developer/admin | 开发人员点击"标记已修复" |
| Fixed | Verified | tester/admin | 测试人员点击"确认完成" |
| Fixed | InProgress | tester/admin | 测试人员点击"修复未通过" |
| Verified | - | - | 流程结束，不可变更 |
